# Relationship Scanner Design

**Date:** 2026-03-22
**Status:** Approved

## Summary

Add a static relationship scanner that derives block relationships from source syntax on every commit â€” no AI tokens required. AI-based relationship generation remains available but is gated by configuration. Each relationship tracks how it was produced (`source: static | ai`) so the two pipelines replace only their own edges and never clobber each other.

---

## Motivation

The relationship graph in `index.json` is currently empty. The most valuable relationship types (`depends_on`, `extends`, `implements`, `tests`) are fully derivable from source syntax without AI inference. Running static derivation on every commit costs nothing and keeps the graph current. AI generation is reserved for types that genuinely require semantic reasoning, and is opt-in to avoid unnecessary API costs.

---

## Relationship Types in Scope

The scanner derives four types, chosen because they are statically derivable and directly actionable for code exploration, impact analysis, and review:

| Type | Derivation |
|---|---|
| `depends_on` | Import declarations and named call sites |
| `extends` | Class superclass (`class Foo extends Bar`) |
| `implements` | Class interface clause (`class Foo implements IBar`) |
| `tests` | Imports from non-test files within `*.test.*` / `*.spec.*` / `*_test.*` files |

All other relationship types (`overrides`, `configures`, `validates`, `documents`, `derived_from`) are not generated by the static scanner. They remain available for AI generation or agent annotation via `git why push --relate-to`.

## Source Tracking

Add `source: 'static' | 'ai'` to the `Relationship` type in block annotation frontmatter:

```yaml
relationships:
  - type: depends_on
    target: src/core/fs/writer.ts::writeFile
    source: static
  - type: validates
    target: src/core/types.ts::BlockFrontmatter
    source: ai
```

**Replacement rules:**
- When the static scanner runs, it deletes all `source: static` edges for files it processes and writes fresh ones.
- When AI generation runs, it deletes all `source: ai` edges for files it processes and writes fresh ones.
- Neither pipeline touches the other's edges.
- Existing relationships without a `source` field are treated as `ai` on first migration.
- `git why push --relate-to` always writes `source: ai`, whether called by a human agent or an API-backed pipeline. This preserves functionality for users who use a coding agent without a configured API key.

## Architecture

### New module: `src/core/relationships/scanner.ts`

A `RelationshipScanner` interface that each language plugin implements:

```typescript
interface RelationshipScanner {
  language: string
  scan(filePath: string, source: string, allBlocks: BlockRegistry): RelationshipInput[]
}
```

`BlockRegistry` is a map of `symbolicRef â†’ ParsedBlock` built from all files in the repo before cross-file edges are resolved. This enables the two-pass approach required to resolve named imports to specific blocks.

**Two-pass execution:**
1. Parse all files â†’ build `BlockRegistry`
2. For each file, run its language scanner â†’ produce `RelationshipInput[]` with `source: 'static'`

### Language support

| Language | `depends_on` | `extends` | `implements` | `tests` |
|---|---|---|---|---|
| TypeScript/JS | âś“ (full AST) | âś“ (full AST) | âś“ (full AST) | âś“ (filename convention) |
| Python | âś“ (regex) | âś“ (regex) | â€” (duck typing) | âś“ (filename convention) |
| Go | âś“ (regex) | â€” (no inheritance) | â€” (implicit interfaces) | âś“ (filename convention) |
| Rust | âś“ (regex) | â€” | âś“ (`impl Trait for Type`) | âś“ (`#[cfg(test)]` / `*_test.rs`) |

TypeScript/JS uses the existing `@typescript-eslint/typescript-estree` AST for precise derivation. Other languages use regex against the existing parser plugin output â€” reliable for import syntax and filename conventions, less precise for structural relationships.

**Edge cases skipped silently:** dynamic imports, re-exports, barrel index files, external (node_modules) imports. The scanner does not guess; it only writes relationships it can derive with confidence.

### Integration points

**Post-commit hook (incremental):** The static scanner runs as the first step of the existing resolution pipeline, operating only on files touched in the commit. The incremental filter already used by `runResolutionPipeline` is reused here.

**`git why scan` command (on-demand):** Runs the static scanner across all files in the repo, replaces all `source: static` edges, rebuilds the index. Useful for initial setup, recovery, or after changing scanner logic.

```bash
git why scan                          # full repo scan
git why scan --file src/core/push/index.ts  # single file, for debugging
```

Output: count of files scanned, relationships found, relationships replaced.

## Configuration

In `.why/config.json` (or the existing whytho config loader):

```json
{
  "relationships": {
    "static_scan": true,
    "ai_scan": "off"
  }
}
```

| Option | Values | Default |
|---|---|---|
| `static_scan` | `true \| false` | `true` |
| `ai_scan` | `"off" \| "manual" \| "on_commit"` | `"off"` if no API key; `"manual"` otherwise |

`ai_scan: "manual"` means AI generation only runs when explicitly invoked (command TBD). `ai_scan: "on_commit"` adds it to the post-commit pipeline alongside static scanning.

---

## Index rebuild

After the scanner writes updated relationships to block annotation frontmatter, the existing `buildIndex()` is called to rebuild `index.json`. No changes to the index builder are required â€” it already reads `relationships` from block frontmatter and builds the bidirectional graph.

---

## Migration

On first run after this change, existing block annotations without `source` on their relationships are treated as `source: ai`. No automated migration script is needed; the scanner will write `source: static` edges on the next commit or `git why scan` invocation.
